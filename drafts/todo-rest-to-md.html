Migrating from RST to Markdown

Everywhere

I used to use RST (ReStructured Text) format for all my FOSS documentation, internal note taking and even for Python docstrings.

However, I’ve decided to make my life simpler by using Markdown everywhere.

This change came about around the summer of 2023. Since then all posts on this blog have been in Markdown (yes all FOUR of them).

Essentially the change happened because I’ve improved my workflow for Markdown documents because they are the standard at work. In fact, thinking back, I can’t think of a work project that I’ve ever contributed to that didn’t use Markdown for documentation stored in repositories.

As we’ve worked to improve and upgrade the quality of our work documentation, I’ve also improved my local workflow. I use Frogmouth to render Markdown documents alongside my editor in Vim. We’ve used https://github.com/DavidAnson/markdownlint/ at work to nail down consistency, but I’ve not wired it into my workflow at home or work yet.

However, all this effort, and the fact I’ve finally learned how to do links in Markdown [ ]( ), means that I’m in a much better place to switch over to it.

Making the switch.

I’ve been using pandoc to make RST to MD conversions. It’s not perfect and I have to fix things up by hand afterwards, but it gets the bulk of the document “right”.

pandoc -s -o rc.md release_checklist.rst

The annoying thing has been keeping history for those docs in git. For this I run the following:

  • Make an initial Markdown clone of the document.
  • git mv the existing RST document to its new MD filename.
  • Move the Markdown clone over the top of the markdown document.

Most recently, I’ve converted the Flake8-AAA release checklist document from RST to Markdown. So, working in the /docs directory of that project:

pandoc -s -o /tmp/rc.md release_checklist.rst
  1. move
git mv release_checklist.rst release_checklist.md
  1. Dump the converted file
mv /tmp/rc.md release_checklist.md

Now at this stage, I usually hand-edit the changes and commit the move and edit at the same time, but you might be more cautious and want to commit the move, and then the edits on top.

Issues

At a personal level, I’ve not had any issues using Markdown over RST.

I find it easier to write and easier to render in terminal.

This blog is running the Pelican static site generator and that handles the Markdown just fine.

So I’ll continue using Markdown everywhere.

There might be interesting issues at an organisational level - this Toot was really interesting and mentions pros and cons of using Markdown for technical documentation (I think “within a team” is implied).
